## Workflow

All non-trivial work flows through the `/deliver` command:
1. `/deliver "description"` ‚Äî starts tracked work
2. Pipeline: intake ‚Üí classify ‚Üí decompose ‚Üí **review-plan** ‚Üí execute ‚Üí verify ‚Üí **evaluate** ‚Üí deliver
3. Every edit is tracked via beads (`.forge/beads/`)
4. Mechanical verification (typecheck, lint, test) runs after execution
5. Tri-agent evaluation runs after verification ‚Äî three independent evaluator agents score the implementation
6. If evaluation score is below threshold, the pipeline loops back to execute with revision instructions

## Spec Ingestion

For large specs, PRDs, or requirements documents:
1. `/ingest <spec-id>` ‚Äî parse a spec into epics, features, and atomic tasks
2. Multi-pass analysis: extract ‚Üí map domains ‚Üí decompose ‚Üí identify skills
3. Review and refine the plan before writing any code
4. Generate custom skills for repetitive patterns
5. Execute phase-by-phase through `/deliver`

Specs are stored in `.forge/specs/`. Run `forge ingest <file>` to add a new spec.

## Quality Gates

All code must pass before delivery:

### Mechanical Verification (pass/fail)
- **Typecheck**: `go vet ./...`
- **Lint**: `golangci-lint run`
- **Test**: `go test ./...`
- **Browser smoke test**: Playwright visual regression at mobile (375x812) + desktop (1440x900) ‚Äî runs automatically when frontend files change

### Tri-Agent Evaluation (scored, 0.0‚Äì1.0)
Three independent evaluator agents review every implementation after mechanical checks pass:
- **Edgar** (`.forge/agents/edgar.md`) ‚Äî Adversarial edge case analysis. Finds what will break in production: robustness, error handling, security surface, brittleness.
- **Code Quality** (`.forge/agents/code-quality.md`) ‚Äî Architecture fit, maintainability, performance, correctness beyond what tests catch.
- **Um-Actually** (`.forge/agents/um-actually.md`) ‚Äî Cross-references implementation against official documentation, framework conventions, and known pitfalls.

Composite score = (Edgar * 0.35) + (Code Quality * 0.35) + (Um-Actually * 0.30). Must reach 0.7 to pass. If below threshold, the pipeline generates a revision brief and loops back to execution (max 3 iterations).

### Sprint Contract Review
Before execution begins, the decomposition plan is reviewed for completeness, file conflict safety, dependency correctness, and verification adequacy. This prevents wasted execution cycles on flawed plans.
